# 超大规模画布高性能方案（10k+ 图文/视频卡片）

目标场景：单个画布项目包含上万张图文 / 视频 / 文字卡片（以及随之而来的数千到
上万条连线），在平移、缩放、拖拽、连线等所有交互下保持 60fps，初始可交互时间
与内存占用不随卡片总量线性增长。

本文是该目标的整体技术方案，作为后续实施的设计输入。方案分层展开，
按依赖关系排出实施顺序；每一层都标明「axle 框架侧」与「应用侧」的分工。

> 修订记录：v2 依据深度 review 重构了三处设计——LOD 结构切换收归窗口化层的
> 预算队列（禁止逐卡函数 child 绑全局档位 atom）；dot 档从「逐卡 host +
> 可选聚合」改为「单个自定义绘制节点」并进入核心路径；持久位置事实源从引擎
> 反转回 model（修订 04 号文档结论的适用边界）。并补齐了 z-order 契约、
> 连线虚拟化、手势中挂载策略、选中集与编辑器集成四个缺失面。

## 0. 现状与瓶颈定位

Phase 1/2 的架构已经把响应式层做到了正确的形状（见
[04-reactive-leafer-state.md](./04-reactive-leafer-state.md)）：

- 属性更新成本 = O(实际变化的绑定数)，与画布总节点数无关；
- 缩放/平移走 leafer zoomLayer 单矩阵，零响应式开销；
- `RxList` patch 映射为最小场景图操作。

因此**规模化的瓶颈不在响应式层，而在「全量挂载」**。以 POC 的卡片结构估算
（`playground/canvas.tsx`：group + 底 rect + 媒体 box + image/canvas + badge +
标题条 + 2 个 text + 4 个 port ≈ 13 个节点/卡）：

| 卡片数 | 场景图节点数 | 后果 |
| ------ | ------------ | ---- |
| 100    | ~1,300       | 舒适区间 |
| 10,000 | ~130,000     | 初始挂载秒级卡死；布局/包围盒全量计算；内存爆炸 |

具体瓶颈拆解（按交互路径）：

1. **初始挂载**：`RxList` 全量渲染 → 10k 次组件函数执行 + 130k 次
   `UICreator.get` + leafer 全量 layout。主线程长任务，白屏数秒。
2. **缩放到全局视野**：leafer 的局部渲染（`usePartRender`）在「变化区域覆盖
   全部可见内容」时退化为全量重绘。10k 卡片同屏时每帧要遍历 + 绘制所有节点，
   阴影 / 半透明描边 / 文本布局是实测重灾区（leafer-ui issue #579）。
3. **命中检测**：leafer 按包围盒树自顶向下剪枝，但卡片层是**一个扁平 group
   挂 10k 子节点**——该 group 的包围盒覆盖整个世界，剪枝失效，每次 pointer
   事件都要线性扫 10k 子节点。4 万个常驻 port（ellipse）同理。
4. **局部渲染的相交遍历**：拖动一张卡片时，partRender 要在全部子节点里找
   与脏区相交者，同样被扁平结构拖成 O(n)/帧。
5. **媒体资源**：每个视频卡片一个独立 rAF + `forceRender()`，每张图片一份
   全尺寸解码位图常驻内存。100 个视频 = 100 个 rAF 循环 + 每帧 100 次渲染
   管线触发；10k 张图 = 数 GB 级解码内存。
6. **连线**：万级卡片对应数千到上万条连线，每条是一个带响应式 `path` 绑定的
   可命中 path 节点，全量挂载的成本结构与卡片相同。

结论：核心策略是**让所有成本从 O(总卡片数) 变成 O(视口内容量)**。
响应式层已经满足这个不变量，下面把挂载、渲染、命中、媒体四条路径也拉齐。

## 1. 数据层前提：持久事实源反转回 model

这是虚拟化的**前置条件**，先于一切机制。

04 号文档确立的「引擎是唯一事实源」（位置不存 model atom，经 `RxUIPosition`
流入响应式世界）在全量挂载下成立，但与虚拟化根本冲突：

- **未挂载的卡片没有引擎对象**，协同编辑、自动布局、程序化移动屏外卡片时
  「引擎事实源」无处落地；
- `RxUIPosition` 在 ref 摘除时把 value 置 `null`（`reactiveLeaferState.ts`
  的 `abort` 实现），下游靠 `?? 初始值` 兜底——「卸载前回写 model」与
  「置 null」的先后取决于 Host 销毁顺序，是脆弱的时序依赖；
- 空间索引需要**全部**卡片的包围盒，只能以 model 为源。

修订后的分工：

- **model 是唯一持久事实源**：`CardModel.position` 是普通
  `atom<IPointData>`，创建即有值、永远非 null。连线、空间索引、序列化
  一律读 model。
- **引擎只在挂载期间代理交互**：卡片挂载时用桥接工具（`RxUIPosition` 的
  变体，暂名 `bindEnginePosition(ui, positionAtom)`）把 `property.change`
  单向写进 model atom；卸载时拆桥，atom 保留最后值。拖拽热路径的性质不变
  （引擎移动元素，响应式层只消费结果），04 号文档的性能结论仍然成立，
  只是把「事实源」的适用边界收窄为**挂载期间的交互热路径**。
- 04 号文档需要补一段适用边界说明，指向本文（配套改动，见实施顺序表）。

## 2. 视口虚拟化（收益最大，框架核心投入）

**原则：场景图里只存在「视口 + 缓冲区」内的卡片，其余卡片只存在于数据层。**

视口 1920×1080、卡片 240×216、常用缩放下同屏全细节卡片约 40–60 张；
加 1.5–2 倍缓冲区后挂载量约 100–200 张（~2,000 节点）——不论总量是
1 万还是 10 万，场景图规模恒定在 leafer 舒适区间。

### 2.1 空间索引（axle 新模块 `spatialIndex`）

- 数据结构：R-tree（推荐 `rbush`）或均匀网格（卡片尺寸相近时网格更简单、
  增量更新 O(1)）。先做网格，接口留出替换空间。
- 索引内容：卡片 `id → bounds`（页面坐标包围盒）+ **连线条目**（见 §5），
  另维护 `cardId → edgeIds` 邻接表。
- 增量维护：**订阅 model position atom**（不是引擎事件——对未挂载卡片
  引擎事件不存在）。拖拽中挂载卡片的引擎变更经 §1 的桥写入 model atom，
  索引自然跟上；增删卡片跟随 `RxList` patch。
- 索引是纯数据层对象，不参与渲染，不在任何交互热路径上做全量重建。

### 2.2 视口窗口化列表（axle 新模块，暂名 `rxWindowedList`）

```ts
const windowed = rxWindowedList(cards, {
  viewport: rxViewport,          // RxViewport（已有）
  lod: rxLod,                    // 档位 atom（见 §3），档位是窗口化的输入
  getBounds: (card) => index.get(card.id),
  buffer: 0.75,                  // 视口外扩比例
  hysteresis: 0.25,              // 滞后带：移出 buffer + hysteresis 才卸载
  pin: (card) => pinned.has(card.id), // 强制保活：拖拽中/选中/播放中
})
// windowed.rows: RxList<{ card: CardModel; lod: 'full' | 'simple' }>
<group>{windowed.rows.map((row) => <Card card={row.card} lod={row.lod} />)}</group>
```

实现要点：

- 订阅 `RxViewport` 与档位 atom，用空间索引查询扩展视口内的 id 集合，与
  上一帧集合 diff，产出**每帧合并为一批的 splice**（进出视口的卡片在列表中
  散落，实际是 O(变化数) 次 splice；框架侧预留 `RxList` 批量 patch 的优化
  口子）。下游复用现有 `RxListHost` 增量机制，不需要新的 Host 类型。
- **档位是行身份的一部分**（`{ card, lod }`）：跨 full↔simple 档位时，
  窗口化层把「重挂载」任务放进同一预算队列分帧执行，旧档位的行保持显示
  直到被替换（不闪空）。**这是 LOD 结构切换唯一合法的通道**，理由见 §3.2。
- **滞后带（hysteresis）**：进入阈值和退出阈值分开，避免卡片在缓冲区边界
  抖动导致反复挂载/卸载。
- **按帧节流 + 挂载预算**：视口变化在 rAF 边界合并；快速平移时一帧可能新进
  几百张卡片，按「每帧最多挂载 N 张（如 20）」时间切片，优先挂载离视口
  中心近的，其余排队。缓冲区的存在保证排队不产生视觉空洞。
- **pin 语义**：拖拽中、选中集、正在播放的视频等必须保活，即使移出缓冲区。
  pin 判定基于 id 集合（见 §8），不持有元素引用。

### 2.3 z-order 契约

虚拟化后兄弟节点次序 = 挂载次序：先滚出再滚回的卡片会被 append 到末尾，
叠放关系随浏览路径漂移——这是必然出现的正确性 bug（卡片重叠是画布常态）。

契约：**`rxWindowedList` 不承诺输出次序，视觉叠放次序由显式 z 决定**。
model 携带稳定的 `zOrder`（如创建序号），卡片 group 绑定 leafer 的
`zIndex={card.zOrder}`；置顶/置底操作改 model 字段。连线层整层在卡片层
下方，层内次序不敏感，无需逐条 z。

### 2.4 与「组件只执行一次」模型的关系

虚拟化意味着卡片组件会被反复销毁/重建：

1. **状态外置是硬约束**：卡片的一切可变状态放在 model（§1 已把位置收归
   model），组件函数只是 model → 场景图的投影。这本来就是 axle 的推荐模式。
2. **槽位复用（第二阶段优化，先不做）**：如果 profile 显示重建成本
   （组件函数执行 + 节点创建）在快速平移时仍是瓶颈，可引入虚拟列表经典的
   节点池方案——槽位组件持有 `slot = atom<CardModel>`，所有绑定读
   `() => slot().title`，换卡片只写一次 atom，细粒度绑定自动刷新全部属性。
   不需要框架改动，是应用层模式；先用「销毁/重建 + 挂载预算」验证，
   实测不够再上。

## 3. LOD 细节分级（缩放全局视野的解药）

虚拟化解决「视口内容量恒定」，但**缩小视野时视口内卡片数会膨胀**
（scale = 0.1 时同屏可能有上万张）。解药是细节分级，且不同档位的
**实现形态不同**：

| 档位 | scale 区间 | 实现形态 | 视口内成本 |
| ---- | ---------- | -------- | ---------- |
| `full`  | > 0.5     | 逐卡 host：完整卡片（媒体 + 文本 + 交互） | ~13 节点/卡 × ~50 卡 |
| `simple`| 0.2 – 0.5 | 逐卡 host：圆角 rect + 缩略图 + 标题（无阴影/描边/port/描述） | ~3 节点/卡 × 数百卡 |
| `dot`   | < 0.2     | **单个自定义绘制节点（DotLayer），不逐卡挂载** | 1 个节点 |

- **阴影必须从 `full` 以下档位去掉**：leafer 阴影是实测最贵的单项特性
  （issue #579 的结论），`full` 档也建议用预渲染九宫格图片替代实时阴影。
- 档位边界加迟滞（如 full↓0.5 / full↑0.55）避免在阈值附近来回抖动。

### 3.1 档位 atom（axle 侧：`RxViewport` 派生）

`RxViewport.value` 在缩放手势中每帧都变，必须把连续 scale 离散化为档位
atom，只有跨档位才触发下游：

```ts
const rxLod = rxLodLevel(rxViewport, { full: 0.5, simple: 0.2 }) // Atom<'full'|'simple'|'dot'>
```

### 3.2 结构切换的唯一通道：窗口化预算队列

**禁止在卡片组件内部用全局档位 atom 的函数 child 切换结构**
（`<>{() => lod() === 'full' ? <FullCard/> : <SimpleCard/>}</>` 是反模式）。
原因：`FunctionHost` 的重建走 `DeferredBindingEffect` 微任务——全局档位
atom 一翻转，视口内**所有**卡片的 FunctionHost 在同一个微任务 flush 里
同步串行重建（销毁旧子树 + 重执行组件 + 重建节点），完全绕过挂载预算，
在 simple 档视口内有数百张卡时是一次确定性的主线程长任务。

正确通道是 §2.2：档位作为 `rxWindowedList` 的输入、行身份的一部分，
档位翻转表现为「经过预算队列、视口中心优先的整批重挂载」，旧行保持显示
直到被替换。卡片组件收到的 `lod` 是**创建时固定的 prop**，组件内部不再
有档位分支。

组件内部允许绑定档位 atom 的只有**属性级**变化（如某个装饰的 `visible`、
阴影参数），这类绑定走 `BindingEffect` 属性写入，成本是 O(绑定数)、
无结构重建，安全。

### 3.3 DotLayer：dot 档的单节点自绘

scale < 0.2 时逐卡 host 没有存在价值：卡片屏幕尺寸 < 50px，媒体/文本不可读，
leafer 对 < 10px 的元素本来就退化为 boxBounds 命中。若仍逐卡挂载，10k 卡
全部进入视口 → 10k 个 host 经每帧 20 张的预算进场需要 500 帧 ≈ 8 秒，
且节点数直接击穿验收标准。

方案：dot 档整层切换为**一个**自定义 UI 节点（leafer 自定义元素，重写
`__draw` / `__updateBoxBounds`）：

- 每帧从**空间索引**查询可见卡片，命令式绘制色块（accent 色小矩形），
  一次遍历、零挂载、零 host 开销；
- 超高密度时在 `__draw` 内部按索引网格聚合为密度块（一格一矩形，颜色深浅
  代表数量）——聚合是 DotLayer 的内部绘制策略，**属于核心路径**，
  不是可选项；
- 该层所在 leafer 关闭局部渲染（官方对「始终整体重绘的层」的建议），
  或把 boxBounds 定义为视口；
- 命中：DotLayer 自身 `hittable={false}`；dot 档的点击（如点击跳转/框选）
  用空间索引反查，不走 leafer 命中；
- 连线在 dot 档并入 DotLayer 绘制（简化为直线段）或整层隐藏（可配置）。

档位进出 dot 的切换 = 挂/卸 DotLayer 这一个节点 + 窗口化列表清空/重建
（simple 行经预算队列进场）。这同时消掉了「跨档位重建风暴」和
「dot 档节点数爆炸」两个问题。

## 4. 交互中降级（缩放/平移过程的帧预算）

缩放/平移**进行中**和**静止后**的渲染预算完全不同。方案：

- axle 侧新增 `RxViewportInteracting`（`RxLeaferState` 子类）：视口手势
  开始（`MoveEvent`/`ZoomEvent`/滚轮）置 `true`，静止 debounce（~150ms）
  后置 `false`。
- **手势进行中冻结窗口扩张**：`rxWindowedList` 在 interacting 期间只处理
  卸载与 pin、**不新挂载**（预算置 0）。短平移由缓冲区兜底不露空洞；
  长距离平移/缩小露出的空白由背景 + DotLayer 兜底（DotLayer 读索引自绘，
  不受挂载预算约束，任何时刻都能画出全部色块）；手势结束后按预算补齐。
  这条策略把「缩小手势同时触发视口集合膨胀 + 跨档位」的最坏路径拆解为
  「手势中只画廉价内容，静止后分帧补细节」。
- 交互中暂停视频帧绘制（共享 ticker 直接 return）、暂停图片高清升档。
- 背景网格层放独立 leafer 层并关闭局部渲染（leafer 官方建议）。

## 5. 连线（edges）的虚拟化

连线不是卡片的附属，需要成体系的对等处理：

- **独立索引条目**：每条连线在空间索引中有自己的包围盒——由两端 port 点
  与贝塞尔控制点合成（`wirePath` 的控制点由端点和 side 推出，O(1) 计算）。
  **不能按端点可见性推导连线可见性**：两端卡片都在屏外、线体穿过视口是
  长连线的常见形态。
- **窗口化**：连线可见 = 其包围盒与扩展视口相交，走与卡片相同的
  `rxWindowedList` 机制（独立实例，输出连线行）。
- **增量维护**：拖动卡片时经 `cardId → edgeIds` 邻接表逐帧更新关联连线的
  索引条目；连线端点位置读 model position atom（§1 保证永远非 null，
  端点卡片挂没挂载都成立）。
- **LOD**：`full`/`simple` 档正常渲染（simple 档细线 + `hittable={false}`）；
  `dot` 档并入 DotLayer 或整层隐藏。
- **命中**：浏览模式整个连线层可配 `hittable={false}`；编辑模式仅 `full`
  档开启 `hitStroke`。

## 6. 场景图组织与命中检测

### 6.1 空间分桶（chunking，兜底优化）

卡片层可按页面坐标网格（如 2048×2048 一格）分桶，每格一个 group：命中
检测与 partRender 的相交遍历按格子剪枝。虚拟化之后场景图本来就小，
分桶只在「大 buffer / simple 档高密度」下有收益，优先级低。

### 6.2 命中面收敛

- **`hittable={false}` 用满**：媒体 image/canvas、badge、标题/描述 text、
  装饰 rect 全部不可命中——每卡只留卡片 group（拖拽/选中）参与命中。
  这同时避免 leafer 为这些节点创建 hitCanvas。
- **port 按需挂载**：POC 的 4 个 port 常驻可命中 ellipse，改为 hover 卡片
  时才挂载 port 层（`RxUIHovered` 驱动函数 child——单卡范围的函数 child
  结构切换是合法的，它只影响一张卡，与 §3.2 禁止的全局档位切换不同）。

## 7. 媒体生命周期（视频/图片的资源预算）

### 7.1 视频

- **可见 + 档位门控**：只有「在视口内 && lod === full && 允许自动播放」的
  视频卡片才创建 `<video>` 元素并拉流；离开视口或降档立即暂停并释放，
  卡面回退到封面帧（poster 图，走图片管线）。
- **全局并发上限**：同时活跃的视频 ≤ N（建议 4–6），超出的显示封面 +
  播放按钮。用全局登记表（id 集合 + atom 计数）实现。
- **单一全局 ticker**：所有活跃视频共享一个 rAF 循环，循环里逐个
  `drawImage` + `forceRender()`，替代每卡一个 rAF。axle 侧沉淀
  `createSharedTicker()` 工具。视频帧率 ≤ 30fps，ticker 可隔帧绘制。

### 7.2 图片

- **缩略图金字塔**：按 lod 档位换 url（dot 不加载、simple 用 ~128px、
  full 用媒体区 2x ≈ 420px；原图只在双击放大时加载）。档位是挂载时固定的
  prop（§3.2），url 随行重挂载自然切换，无需响应式绑定档位。
- **解码预算队列**：集中调度 `createImageBitmap`，按「距视口中心距离」
  排优先级，每帧限量解码，快速平移时队首过期任务直接丢弃。
- **卸载释放**：卡片卸载即释放位图引用；浏览器缓存负责二次加载的网络层，
  内存只为视口内容付费。

## 8. 选中集与编辑器集成

虚拟化后元素会被销毁重建，一切「持有元素引用」的机制都要改为 id-based：

- **选中集是数据层的 id 集合**（`RxSet<id>` 或等价物），不持有 leafer
  元素引用。选中的卡片进入 `rxWindowedList` 的 pin 集合强制保活。
- **框选/全选在数据层做**：框选 = 空间索引范围查询（含屏外卡片），
  Ctrl+A = 全量 id；选中反馈只对挂载中的卡片渲染（选中框绑定
  `() => selected.has(card.id)`）。
- **leafer-editor 的 Editor 与虚拟化存在结构性冲突**（Editor 持元素引用画
  变换框，被选元素卸载时编辑框悬空）。Phase 3 集成时二选一：选中元素
  一律 pin（小选中集可行，「全选 10k 张再缩到 dot 档」需上限保护）；
  或不用 Editor 的元素绑定模式，选框/变换手柄基于数据层自绘。
  本方案先按前者设计 pin 上限（如 200，超出退化为只读框选反馈）。

## 9. 数据层与启动路径

- **数据加载**：10k 卡片的 model 数据（含位置）先行加载并建索引
  （网格索引 O(n)，一次性 < 10ms 量级），**不创建任何 Host**；首帧只挂载
  视口内卡片 → 可交互时间与总量无关。
- `RxList` 全量数据常驻没有问题（1 万个普通对象 + position atom 是 MB 级）。
  引擎桥（§1）只在挂载期间存在，监听成本天然 O(视口内容量)。
- 超大项目（10 万+）再考虑数据分页/按区域拉取，接口在空间索引上天然对齐
  （按格子拉取），本方案不展开。

## 10. 度量与验收

没有基线的优化不可信。配套建设：

1. **压测 playground**（`playground/stress.tsx`）：参数化生成 1k / 10k / 50k
   卡片 + 按比例的连线（图文/视频混合），复用 POC 的卡片组件。
2. **指标面板**：FPS（rAF 间隔滑动窗口）、场景图节点数、挂载/卸载次数、
   JS 堆（`performance.memory`）。
3. **验收标准**（10k 卡片，中端设备）：
   - 平移/缩放全程 ≥ 55fps（含跨档位瞬间，不允许单帧 > 100ms 的长任务）；
   - 拖动单卡 + 10 条关联连线 ≥ 55fps；
   - 首次可交互 < 1s（数据就绪后）；
   - 场景图节点数峰值 < 5,000（dot 档 = DotLayer 单节点，该标准在全档位
     下自洽）；
   - 快速平移 5 屏后无视觉空洞残留 > 200ms（手势中由 DotLayer/背景兜底）；
   - 卡片叠放次序在任意浏览路径后保持稳定（z-order 回归用例）。
4. `Debug.showRepaint = true` 验证脏区不被意外放大（阴影、滤镜会扩大
   renderBounds）。

## 12. 明确不做的方向（及原因）

- **换 WebGL 渲染器**：leafer 架构上可插拔，但当前瓶颈全部在「画多少」而非
  「怎么画」。虚拟化 + LOD 后视口内节点数恒定在数百级，canvas 2d 足够；
  dot 档的万级色块由 DotLayer 单节点命令式绘制，同样不需要 GPU 管线。
  留作 leafer 引擎层未来选项，不进本方案。
- **Web Worker 渲染**：leafer 支持 worker，但跨线程后引擎桥/事件桥接复杂度
  陡增，且主线程瓶颈同样被虚拟化消解。不进本方案。
- **双向绑定视口**：维持 Phase 2 结论，视口反向操作走 `leafer.zoom()`。
- **全量 bitmap 缓存整个画布**（tile 化 minimap 除外）：缓存失效管理复杂，
  与 LOD 收益重叠。若后续 leafer 官方「光速引擎」插件成熟，届时再评估。
