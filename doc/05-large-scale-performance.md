# 超大规模画布高性能方案（10k+ 图文/视频卡片）

目标场景：单个画布项目包含上万张图文 / 视频 / 文字卡片（以及随之而来的数千到
上万条连线），在平移、缩放、拖拽、连线等所有交互下保持 60fps，初始可交互时间
与内存占用不随卡片总量线性增长。

本文是该目标的整体技术方案，作为后续实施的设计输入。方案分层展开，
按依赖关系排出实施顺序；每一层都标明「axle 框架侧」与「应用侧」的分工。

### 热路径常数收敛（post-v6 优化批次）

核心路径验收达标后的一轮常数因子收敛，行为契约不变（全部有测试覆盖）：

- **窗口化增量判定**：触发源 3（索引 write-through 变更）不再触发全量
  重算——只对变更条目基于缓存窗口做进出判断并修补队列（含撤销反向
  任务）。拖拽帧上「卡片 + 邻接连线逐帧更新索引」从
  「全量查询 + diff + 三次排序/帧」降为 O(变更条目数)；
- **队列游标化 + 挂载批量 splice**：预算队列用游标消化（替代 O(n) 的
  shift），同帧多个挂载合并为一次多行 splice（批 4，减少 trigger 派发）；
  卸载/替换的行定位用下标提示 Map（均摊 O(1)，替代 rowIds.indexOf）；
  增量修补路径的队列数组有滞留上界（撤销只删集合、push 只增数组，长纯
  增量会话——手势中冻结挂载 + 条目反复进出窗口——会无界累积废任务；
  剩余段长超过「有效数 ×2 + 64」时按集合过滤压实，均摊 O(1)/次修补，
  同 id 保留首个有效条目、与出队语义一致）；
- **空间索引**：`forEachIn` 用「主 cell clamp 判定」做零分配去重（删掉
  每查询一个 seen Set）；`forEachCell` 的主格计数改为 set/delete 时增量
  维护，dot 档密度聚合重绘从 O(视口内条目数) 降为 O(相交 cell 数)；
- **引擎桥**：`bindPosition`（单 effect 写双轴，no-op 求值减半）与
  `coalesce`（同帧 x/y 轴事件合并为一次微任务写穿，下游连线/索引/底衬
  失效每帧减半）两个可选项；
- **DotLayer**：脏区从单一并集改为至多 8 个矩形（相交**级联**合并——
  并集可能新覆盖此前不相交的脏区，必须吸收到收敛为止，否则同一区域
  会被重复失效；溢出并入增长最小者后同样再级联一轮），脏区列表保持
  两两不相交，协同场景多点变更不再撑成整层重绘；绘制循环缓存
  fillStyle，同色连续块只做一次状态切换；
- **Host 层**：AtomHost/FunctionHost 与各自的绑定 effect 合并为同一对象
  （每绑定省一个对象 + 一个闭包）；ComponentHost 的 effect/回调集合惰性
  分配，render 后销毁自己的占位符（区间委托给 innerHost，每组件少一个
  常驻场景图节点，02 号文档契约已同步修订）；
- **杂项**：`insertBefore` 尾部锚点快速路径（窗口化挂载的主路径不再
  线性扫 children）；`RxViewportInteracting` 改惰性单定时器（手势帧不再
  每事件一对 clearTimeout/setTimeout）；压测页连线 path 改 leafer 命令
  数组（省字符串构造 + 引擎解析）。
- **配套**：`scripts/stress-smoke.mjs`（headless Chrome 对压测页的
  端到端冒烟，覆盖 §10 验收标准的自动化子集）。

## 0. 现状与瓶颈定位

Phase 1/2 的架构已经把响应式层做到了正确的形状（见
[04-reactive-leafer-state.md](./04-reactive-leafer-state.md)）：

- 属性更新成本 = O(实际变化的绑定数)，与画布总节点数无关；
- 缩放/平移走 leafer zoomLayer 单矩阵，零响应式开销；
- `RxList` patch 映射为最小场景图操作。

因此**规模化的瓶颈不在响应式层，而在「全量挂载」**。以 POC 的卡片结构估算
（`playground/canvas.tsx`：group + 底 rect + 媒体 box + image/canvas + badge +
标题条 + 2 个 text + 4 个 port ≈ 13 个节点/卡）：

| 卡片数 | 场景图节点数 | 后果                                            |
| ------ | ------------ | ----------------------------------------------- |
| 100    | ~1,300       | 舒适区间                                        |
| 10,000 | ~130,000     | 初始挂载秒级卡死；布局/包围盒全量计算；内存爆炸 |

具体瓶颈拆解（按交互路径）：

1. **初始挂载**：`RxList` 全量渲染 → 10k 次组件函数执行 + 130k 次
   `UICreator.get` + leafer 全量 layout。主线程长任务，白屏数秒。
2. **缩放到全局视野**：leafer 的局部渲染（`usePartRender`）在「变化区域覆盖
   全部可见内容」时退化为全量重绘。10k 卡片同屏时每帧要遍历 + 绘制所有节点，
   阴影 / 半透明描边 / 文本布局是实测重灾区（leafer-ui issue #579）。
3. **命中检测**：leafer 按包围盒树自顶向下剪枝，但卡片层是**一个扁平 group
   挂 10k 子节点**——该 group 的包围盒覆盖整个世界，剪枝失效，每次 pointer
   事件都要线性扫 10k 子节点。4 万个常驻 port（ellipse）同理。
4. **局部渲染的相交遍历**：拖动一张卡片时，partRender 要在全部子节点里找
   与脏区相交者，同样被扁平结构拖成 O(n)/帧。
5. **媒体资源**：每个视频卡片一个独立 rAF + `forceRender()`，每张图片一份
   全尺寸解码位图常驻内存。100 个视频 = 100 个 rAF 循环 + 每帧 100 次渲染
   管线触发；10k 张图 = 数 GB 级解码内存。
6. **连线**：万级卡片对应数千到上万条连线，每条是一个带响应式 `path` 绑定的
   可命中 path 节点，全量挂载的成本结构与卡片相同。

结论：核心策略是**让所有成本从 O(总卡片数) 变成 O(视口内容量)**。
响应式层已经满足这个不变量，下面把挂载、渲染、命中、媒体四条路径也拉齐。

## 1. 数据层前提：持久事实源反转回 model

这是虚拟化的**前置条件**，先于一切机制。

04 号文档确立的「引擎是唯一事实源」（位置不存 model atom，经 `RxUIPosition`
流入响应式世界）在全量挂载下成立，但与虚拟化根本冲突：

- **未挂载的卡片没有引擎对象**，协同编辑、自动布局、程序化移动屏外卡片时
  「引擎事实源」无处落地；
- `RxUIPosition` 在 ref 摘除时把 value 置 `null`（`reactiveLeaferState.ts`
  的 `abort` 实现），下游靠 `?? 初始值` 兜底——「卸载前回写 model」与
  「置 null」的先后取决于 Host 销毁顺序，是脆弱的时序依赖；
- 空间索引需要**全部**卡片的包围盒，只能以 model 为源。

修订后的分工：

- **model 是唯一持久事实源**：`CardModel.position` 是普通
  `atom<IPointData>`，创建即有值、永远非 null。连线、空间索引、序列化
  一律读 model。
- **引擎只在挂载期间代理交互**：卡片挂载时用桥接工具（`RxUIPosition` 的
  变体，暂名 `bindEnginePosition(ui, positionAtom)`）把 `property.change`
  单向写进 model atom；卸载时拆桥，atom 保留最后值。拖拽热路径的性质不变
  （引擎移动元素，响应式层只消费结果），04 号文档的性能结论仍然成立，
  只是把「事实源」的适用边界收窄为**挂载期间的交互热路径**。
- **挂载期间的完整接线**（实现必须按此接，不能只取初始值）：卡片 group 的
  `x`/`y` **绑定** model atom（`x={() => card.position().x}`）——只有这样，
  程序化 / 协同远程移动一张**已挂载**的卡片（写 atom）才会经绑定作用到引擎。
  由此形成的环按如下方式终止：
  - 引擎 → atom：桥带 `shallowEqual` 守卫，位置没变不写 atom；
  - atom → 引擎：拖拽帧上「引擎动 → 桥写 atom → 绑定以同值回写 `ui.x`」，
    该回写被 leafer 属性 setter 的等值检查吸收为 no-op，环在一跳内终止。
    这正是 04 号文档当年刻意绕开的双向环——事实源反转后环不可避免，
    靠「写前守卫 + 等值 no-op」两道闸终止，需要单测覆盖（拖拽帧无第二次
    `property.change`）。已知有界开销：x/y 两个绑定都依赖整个 `{x,y}` atom，
    拖拽帧上桥写 2 次 atom（x/y 双事件，04 号文档已知）× 2 个绑定 =
    每帧 4 次 no-op 求值；实现时可合并为单 effect 或拆分 x/y atom。
- 04 号文档需要补一段适用边界说明，指向本文（配套改动，见实施顺序表）。

## 2. 视口虚拟化（收益最大，框架核心投入）

**原则：场景图里只存在「视口 + 缓冲区」内的卡片，其余卡片只存在于数据层。**

视口 1920×1080、卡片 240×216、常用缩放下同屏全细节卡片约 40–60 张；
加 1.5–2 倍缓冲区后挂载量约 100–200 张（~2,000 节点）——不论总量是
1 万还是 10 万，场景图规模恒定在 leafer 舒适区间。

### 2.1 空间索引（axle 新模块 `spatialIndex`）

- 数据结构：R-tree（推荐 `rbush`）或均匀网格（卡片尺寸相近时网格更简单、
  增量更新 O(1)）。先做网格，接口留出替换空间。
- 索引内容：卡片 `id → bounds`（页面坐标包围盒）+ **连线条目**（见 §5），
  另维护 `cardId → edgeIds` 邻接表。
- **bounds 是 model 的一等公民（硬约束）**：width/height 与位置一样持久化
  在 model 上，索引只读 model，**绝不依赖引擎 layout 的测量结果**——
  未挂载的卡片永远不会被 leafer 测量，靠引擎测量的 bounds 对虚拟化是
  循环依赖。固定尺寸卡片（POC 形态）天然满足；**文字卡片的自动高度**
  必须在创建/编辑内容时测好写回 model，尺寸变化经 write-through 更新
  索引条目。测量的**权威路径**（避免双事实源——手工 `measureText` 折行
  的断词/字距/字体回退规则与 leafer `Text` 排版不保证一致，测出的高度
  与渲染高度有偏差，索引就错了）：
  - **首选：固定行数截断**（标题 1 行 + 描述 N 行省略号，POC 即此形态），
    高度是确定性公式，测量问题整体消失；
  - 确需自由高度时，用**一次性离屏 leafer 测量**（同一排版引擎，无偏差），
    不用手工 `measureText` 折行。
- 增量维护采用 **write-through**，索引本身不做任何响应式订阅：位置写入
  统一收口——引擎桥（§1）在写 model atom 的同时同步更新索引条目及关联
  连线条目（经邻接表）；程序化/协同移动走 `moveCard(id, pos)` 类 helper，
  同样一次写穿 atom + 索引。增删卡片/连线在数据层操作里同步维护索引。
  明确**禁止**两种订阅式实现：给每张卡建常驻 effect（1 万个监听的建立与
  内存成本白付，屏外卡片根本不动），或用一个覆盖全部 atom 的 computed
  （任何一张卡动了整体重算）。
- 索引是纯数据层对象，不参与渲染，不在任何交互热路径上做全量重建。
- **bounds 必须落在设计包络内（写入口断言）**：`set` 对四个字段做
  `Number.isFinite` 检查，并断言两条网格包络——单条目覆盖的 cell 数
  ≤ 2^16（有限但巨大的尺寸同样会让 cell 循环失控：width 1e6 时一次 set
  就是数百万次 Map 操作，1e8 直接挂死，仅 isFinite 防不住）；cell 坐标
  在 ±2^25 编码域内（越界后 cellKey 运算超出 2^53，keyToCx/keyToCy
  逆映射在浮点精度损失下出错，稀疏回退漏报、密度聚合画错格子，全部是
  静默错误）。NaN 会让条目落进 NaN cell、对所有查询永久不可见（卡片
  静默消失且事故点与症状点相隔甚远）。write-through 是唯一写入口，
  事故必须暴露在写入点；断言先于一切簿记写入，失败不留半写入状态。
- **查询成本与查询矩形面积解耦**：`forEachIn` / `forEachCell` 先把 cell
  循环 clamp 进「已占用 cell 包围盒」（grow-only 增量维护），并在查询范围
  覆盖的 cell 数超过实际占用数时回退为遍历占用 cell 表按范围过滤——成本
  上限是 O(占用 cell 数) 而不是 O(矩形覆盖 cell 数)。否则极限缩小视野
  （scale 0.01 时 enterRect 可达数十万 px 宽）会让窗口化重算与 DotLayer
  重绘每帧对几十万个空 cell 做 Map 查询，且随缩小倍数平方增长；查询矩形
  含 Infinity（viewRect 除以异常 scale）时更是无限循环。

### 2.2 视口窗口化列表（axle 新模块，暂名 `rxWindowedList`）

```ts
const windowed = rxWindowedList(cards, {
  viewport: rxViewport,          // RxViewport（已有）
  lod: rxLod,                    // 档位 atom（见 §3），档位是窗口化的输入
  getBounds: (card) => index.get(card.id),
  buffer: 0.75,                  // 视口外扩比例
  hysteresis: 0.25,              // 滞后带：移出 buffer + hysteresis 才卸载
  pin: (card) => pinned.has(card.id), // 强制保活：拖拽中/选中/播放中
})
// windowed.rows: RxList<{ card: CardModel; lod: 'full' | 'simple' }>
// windowed.mountedIds: 当前实际挂载的 id 集合（预算队列消化后的真实状态，
//   与目标集合可能相差在途的排队项）——消费方是 DotLayer 的「跳过已挂载
//   卡片」选项（§3.3）。注意：§8 的选中框绑定按卡片挂载天然只对挂载中的
//   卡片生效，不需要本集合；dot 档底衬选中高亮需要的是 selected 集合。
<group>{windowed.rows.map((row) => <Card card={row.card} lod={row.lod} />)}</group>
```

实现要点：

- **重算触发源必须完整覆盖四类**（漏一类就是正确性 bug）：
  1. 视口变化（`RxViewport`）；
  2. 档位变化（档位 atom）；
  3. **空间索引变更**——含数据层增删卡片/连线（视口内新建的卡片必须立刻
     挂载）、未挂载卡片被程序化/协同移入或移出视口、以及**拖拽中经邻接表
     逐帧更新的连线包围盒**（拖动卡片把一条线甩进视口时，这条线必须在
     拖拽帧上挂载，不能等下一次视口变化）。索引在 write-through 写入时
     向窗口化列表发变更通知（普通回调即可，同帧合并）；
  4. pin 集合变化。
     以上全部在 rAF 边界合并为一次重算。**`buffer` / `hysteresis` getter 的
     响应式依赖同样在追踪集内**：它们决定窗口本身（enterRect / keepRect），
     getter 依赖档位之外的响应式输入（如设置项 atom）时，写入也必须触发
     全量重算——只在重算时读取会漏掉这类触发（重算运行在 rAF 回调里，
     不被任何 effect 追踪）。
- 重算用空间索引查询扩展视口内的 id 集合，与上一帧集合 diff，产出
  **每帧合并为一批的 splice**（进出视口的卡片在列表中散落，实际是
  O(变化数) 次 splice；框架侧预留 `RxList` 批量 patch 的优化口子）。
  下游复用现有 `RxListHost` 增量机制，不需要新的 Host 类型。
- **行 diff 按 `(card.id, lod)` 复合键**，不按对象身份——行元组每次重算
  都是新对象，若按身份 diff，视口微动就会被判成整批替换，虚拟化直接退化
  为全量重建。键相同的行必须复用既有 Host（splice 不触碰它们）。
- **档位是行身份的一部分**（`{ card, lod }`）：跨 full↔simple 档位时，
  窗口化层把「重挂载」任务放进同一预算队列分帧执行，旧档位的行保持显示
  直到被替换（不闪空）。**这是 LOD 结构切换唯一合法的通道**，理由见 §3.2。
- **滞后带（hysteresis）**：进入阈值和退出阈值分开，避免卡片在缓冲区边界
  抖动导致反复挂载/卸载。
- **按帧节流 + 挂载/卸载双向预算**：快速平移时一帧可能新进几百张卡片，
  时间切片分帧执行，优先挂载离视口中心近的，其余排队；**卸载同样进预算
  队列**（优先卸离视口最远的）——档位切换、手势结束等场景的整批卸载是
  同步 teardown（effect 拆除 + 监听解绑 + 节点移除引发的脏区合并），
  数百个 host 一帧销毁可能单独超出帧预算。缓冲区 + 底衬层（§3.3）保证
  排队不产生视觉空洞。预算队列的三条执行语义：
  - **计量单位是「行操作」**，预算按**帧时间自适应**（如每帧结构操作
    ≤ 4ms，实测校准），不用固定张数——固定「20 张/帧」在 simple 档
    最坏情形（数百张）下补齐时间没有余量，且不同设备差异大；
  - **同卡跨档位替换必须同帧成对执行**（同一批 splice 里 remove +
    insert，替换对计 1 单位）——这是「旧行保持显示直到被替换（不闪空）」
    成立的前提，实现成两个独立队列就会闪空；
  - **优先级：替换 > 新挂载 > 卸载**（卸载最不可见，可以最晚做）。
- **pin 语义**：拖拽中、选中集、正在播放的视频等必须保活，即使移出缓冲区。
  pin 判定基于 id 集合（见 §8），不持有元素引用。**pinned 行在 dot 档的
  形态**：dot 档下行列表清空，但 pinned 行以 `simple` 形态保活（正在拖拽
  的行保持进入 dot 档前的形态直到手势结束）；此时卡片屏幕尺寸只有几个
  像素，选中反馈退化为底衬色块高亮，不再渲染选中框。

### 2.3 z-order 契约

虚拟化后兄弟节点次序 = 挂载次序：先滚出再滚回的卡片会被 append 到末尾，
叠放关系随浏览路径漂移——这是必然出现的正确性 bug（卡片重叠是画布常态）。

契约：**`rxWindowedList` 不承诺输出次序，视觉叠放次序由显式 z 决定**。
model 携带稳定的 `zOrder`（如创建序号），卡片 group 绑定 leafer 的
`zIndex={card.zOrder}`；置顶/置底操作改 model 字段。连线层整层在卡片层
下方，层内次序不敏感，无需逐条 z。

**zIndex 与 Host 树契约的例外声明**：leafer 的 zIndex 是**物理重排**——
`Branch.__updateSortChildren` 会对 children 数组原地 sort（zIndex 0 的
axle 占位符会被排到带 zOrder 的卡片之前）。这使 02 号文档 Host 契约中
「`getNodes()` 顺序与场景图一致」的不变量对绑定了 zIndex 的列表**不再
成立**。分析结论与约束：

- axle 的簿记是引用式的（`insertBefore` 调用时实时 `indexOf(anchor)`，
  行 host 按引用持有、锚点取 `firstNode`），splice 路径可以容忍物理重排；
- **绑定 zIndex 的列表禁止触发 reorder patch**（`RxListHost` 的 LIS 搬移
  假设物理顺序与簿记一致）——窗口化列表只发 splice，天然满足，但必须
  写进契约防止未来误用。**开发期运行时防线**：`setListDiagnostics(true)`
  时 `RxListHost.handleReorder` 检测分支内的 zIndex 参与排序并报告契约
  违例（error 钩子 / `console.error`）——误用只会以视觉叠放错乱出现、
  无从定位，必须在开发期变成显式错误；报告不中断簿记调整（生产路径
  只多一次布尔检查）；
- 「zIndex 重排 + 占位符锚点插入删除」的**兼容性测试**是实施硬性项
  （见实施顺序表），02 号文档同步补例外说明；
- **备选方案**（若测试发现问题的回退路径）：不用 zIndex，窗口化输出按
  zOrder 排序、挂载时按序插入，z 变更 = 一次 remove + insert splice。
  成本与 zIndex 方案相当，代价是 diff 逻辑多维护一个有序性。

### 2.4 与「组件只执行一次」模型的关系

虚拟化意味着卡片组件会被反复销毁/重建：

1. **状态外置是硬约束**：卡片的一切可变状态放在 model（§1 已把位置收归
   model），组件函数只是 model → 场景图的投影。这本来就是 axle 的推荐模式。
2. **槽位复用（第二阶段优化，先不做）**：如果 profile 显示重建成本
   （组件函数执行 + 节点创建）在快速平移时仍是瓶颈，可引入虚拟列表经典的
   节点池方案——槽位组件持有 `slot = atom<CardModel>`，所有绑定读
   `() => slot().title`，换卡片只写一次 atom，细粒度绑定自动刷新全部属性。
   不需要框架改动，是应用层模式；先用「销毁/重建 + 挂载预算」验证，
   实测不够再上。

## 3. LOD 细节分级（缩放全局视野的解药）

虚拟化解决「视口内容量恒定」，但**缩小视野时视口内卡片数会膨胀**
（scale = 0.1 时同屏可能有上万张）。解药是细节分级，且不同档位的
**实现形态不同**：

| 档位     | scale 区间 | 实现形态                                                      | 视口内成本           |
| -------- | ---------- | ------------------------------------------------------------- | -------------------- |
| `full`   | > 0.5      | 逐卡 host：完整卡片（媒体 + 文本 + 交互）                     | ~13 节点/卡 × ~50 卡 |
| `simple` | 0.2 – 0.5  | 逐卡 host：圆角 rect + 缩略图 + 标题（无阴影/描边/port/描述） | ~3 节点/卡 × 数百卡  |
| `dot`    | < 0.2      | 不逐卡挂载，仅由**常驻底衬层 DotLayer**（§3.3）表现           | 底衬层 1 个节点      |

DotLayer 是**全档位常驻**的底衬（不是 dot 档专属开关）：full/simple 档下
已挂载的卡片盖住自己的色块，dot 档下它是唯一表现形态。这保证任何档位、
任何时刻（包括手势中冻结挂载时，见 §4）画布上都不会出现空白。

- **阴影必须从 `full` 以下档位去掉**：leafer 阴影是实测最贵的单项特性
  （issue #579 的结论），`full` 档也建议用预渲染九宫格图片替代实时阴影。
- 档位边界加迟滞（如 full↓0.5 / full↑0.55）避免在阈值附近来回抖动。

### 3.1 档位 atom（axle 侧：`RxViewport` 派生）

`RxViewport.value` 在缩放手势中每帧都变，必须把连续 scale 离散化为档位
atom，只有跨档位才触发下游：

```ts
const rxLod = rxLodLevel(rxViewport, { full: 0.5, simple: 0.2 }) // Atom<'full'|'simple'|'dot'>
```

### 3.2 结构切换的唯一通道：窗口化预算队列

**禁止在卡片组件内部用全局档位 atom 的函数 child 切换结构**
（`<>{() => lod() === 'full' ? <FullCard/> : <SimpleCard/>}</>` 是反模式）。
原因：`FunctionHost` 的重建走 `DeferredBindingEffect` 微任务——全局档位
atom 一翻转，视口内**所有**卡片的 FunctionHost 在同一个微任务 flush 里
同步串行重建（销毁旧子树 + 重执行组件 + 重建节点），完全绕过挂载预算，
在 simple 档视口内有数百张卡时是一次确定性的主线程长任务。

正确通道是 §2.2：档位作为 `rxWindowedList` 的输入、行身份的一部分，
档位翻转表现为「经过预算队列、视口中心优先的整批重挂载」，旧行保持显示
直到被替换。卡片组件收到的 `lod` 是**创建时固定的 prop**，组件内部不再
有档位分支。

组件内部允许绑定档位 atom 的只有**属性级**变化（如某个装饰的 `visible`、
阴影参数），这类绑定走 `BindingEffect` 属性写入，成本是 O(绑定数)、
无结构重建，安全。

### 3.3 DotLayer：常驻底衬层（单节点自绘）

scale < 0.2 时逐卡 host 没有存在价值：卡片屏幕尺寸 < 50px，媒体/文本不可读，
leafer 对 < 10px 的元素本来就退化为 boxBounds 命中。若仍逐卡挂载，10k 卡
全部进入视口 → 10k 个 host 经每帧 20 张的预算进场需要 500 帧 ≈ 8 秒，
且节点数直接击穿验收标准。

方案：**一个**自定义 UI 节点（leafer 自定义元素，重写 `__draw` /
`__updateBoxBounds`），常驻挂载，覆盖全部档位。**层序必须是
背景 < DotLayer < 连线层 < 卡片层**——不能按直觉放在「卡片层正下方」
（即连线层之上）：那样一条经过未挂载卡片区域的连线会被色块盖住，
视觉上凭空断一截。色块放在连线层之下的代价只是 dot 档下连线（若配置
为不隐藏）压在色块上方，可接受：

- 每次绘制从**空间索引**查询可见卡片，命令式绘制色块（accent 色小矩形），
  一次遍历、零挂载、零 host 开销。不感知档位——full/simple 档下已挂载的
  卡片盖住自己的色块，dot 档下色块就是全部。视觉细节：卡片圆角处会露出
  底下色块直角，色块内缩 2px 或按 `windowed.mountedIds`（§2.2）跳过
  已挂载卡片，二选一；
- **失效机制（必须显式接线）**：DotLayer 在 leafer 的「属性变更 → 脏区 →
  重绘」管线**之外**——本地拖拽和视口变化碰巧会产生覆盖它的脏区，但
  **纯数据变更没有任何 leafer 属性变化**（协同/程序化移动未挂载卡片、
  dot 档下增删卡片），不会触发任何重绘，色块会停在陈旧位置。因此
  DotLayer 必须**订阅与窗口化列表同一条索引变更通知通道**（§2.2 触发源
  3，rAF 合并），对变更条目新旧包围盒的并集主动失效（`forceRender` 或
  等价的局部 invalidate）。这是协同场景正确性的必要条件，不是优化；
- **兜底职责**：手势中冻结挂载（§4）、预算队列排队中、长距离跳转后，
  未挂载区域显示的是色块而不是空白；预算补挂的视觉效果是「色块 → 细节
  浮现」。DotLayer 的绘制只依赖索引，不受挂载预算约束；
- 超高密度时在 `__draw` 内部按索引网格聚合为密度块（一格一矩形，颜色深浅
  代表数量）——聚合是 DotLayer 的内部绘制策略，**属于核心路径**，
  不是可选项；
- 成本边界：dot 档整层重绘 = 一次索引查询 + 万级 `fillRect`（毫秒级）；
  full 档下卡片拖动触发的 partRender 脏区会连带重绘底衬被裁剪的部分，
  `__draw` 内按脏区（或视口）查索引，迭代量受查询结果约束，可控。
  若实测干扰局部渲染，把 DotLayer 放独立 leafer 层并关闭该层局部渲染
  （官方对「始终整体重绘的层」的建议）；
- 命中：DotLayer 自身 `hittable={false}`；dot 档的点击（如点击跳转/框选）
  用空间索引反查，不走 leafer 命中；
- 连线在 dot 档并入 DotLayer 绘制（简化为直线段）或整层隐藏（可配置）。

档位进出 dot 的切换不再挂/卸任何节点：只是窗口化列表清空/重建
（simple 行经预算队列进出场），底衬始终在场。这同时消掉了
「跨档位重建风暴」「dot 档节点数爆炸」「兜底层缺位」三个问题。
一个顺带的收益：项目以「适应全部内容」视图打开（常见默认）时直接落在
dot 档，首帧就是一次底衬自绘，可交互时间趋近于零。

## 4. 交互中降级（缩放/平移过程的帧预算）

缩放/平移**进行中**和**静止后**的渲染预算完全不同。方案：

- axle 侧新增 `RxViewportInteracting`（`RxLeaferState` 子类）：视口手势
  开始（`MoveEvent`/`ZoomEvent`/滚轮）置 `true`，静止 debounce（~150ms）
  后置 `false`。
- **手势进行中冻结窗口扩张**：`rxWindowedList` 在 interacting 期间只处理
  卸载与 pin、**不新挂载**（挂载预算置 0）。短平移由缓冲区兜底；长距离
  平移/缩小露出的未挂载区域由**常驻底衬 DotLayer（§3.3）**显示色块——
  底衬在所有档位都在场，这条兜底不依赖当前档位；手势结束后按预算补齐，
  视觉上是「色块 → 细节浮现」。这条策略把「缩小手势同时触发视口集合膨胀 +
  跨档位」的最坏路径拆解为「手势中只画廉价内容，静止后分帧补细节」。
- 交互中暂停视频帧绘制（共享 ticker 直接 return）、暂停图片高清升档。
- 背景网格层放独立 leafer 层并关闭局部渲染（leafer 官方建议）。

## 5. 连线（edges）的虚拟化

连线不是卡片的附属，需要成体系的对等处理：

- **独立索引条目**：每条连线在空间索引中有自己的包围盒——由两端 port 点
  与贝塞尔控制点合成（`wirePath` 的控制点由端点和 side 推出，O(1) 计算）。
  **不能按端点可见性推导连线可见性**：两端卡片都在屏外、线体穿过视口是
  长连线的常见形态。
- **窗口化**：连线可见 = 其包围盒与扩展视口相交，走与卡片相同的
  `rxWindowedList` 机制（独立实例，输出连线行）。**连线的行身份只有
  id、不含 lod**：full↔simple 的差异只有 `strokeWidth` / `hittable` 这类
  属性，按 §3.2 的规则走**属性级档位绑定**即可，不需要跨档位重挂载——
  这也是 §3.2「属性级绑定合法」的正面示例。
- **增量维护**：拖动卡片时经 `cardId → edgeIds` 邻接表逐帧更新关联连线的
  索引条目；连线端点位置读 model position atom（§1 保证永远非 null，
  端点卡片挂没挂载都成立）。
- **LOD**：`full`/`simple` 档正常渲染（simple 档细线 + `hittable={false}`）；
  `dot` 档并入 DotLayer 或整层隐藏。
- **命中**：浏览模式整个连线层可配 `hittable={false}`；编辑模式仅 `full`
  档开启 `hitStroke`。

## 6. 场景图组织与命中检测

### 6.1 空间分桶（chunking，兜底优化，与逐卡 zIndex 互斥）

卡片层可按页面坐标网格（如 2048×2048 一格）分桶，每格一个 group：命中
检测与 partRender 的相交遍历按格子剪枝。

**硬性冲突警告**：leafer 的 `zIndex` 只在**同一父节点的兄弟之间**排序。
分桶后跨 chunk 的两张重叠卡片（卡片骑在格子边界附近是常态），叠放次序由
chunk group 的顺序决定，§2.3 的 `zOrder` 契约完全失效。因此分桶与逐卡
zIndex **互斥**，只有在放弃逐卡 z 序（或接受边界异常）时可用。虚拟化后
挂载量本来就恒定在数百级、本节本来就是低优先级兜底——加上这条互斥，
实施预期进一步降级为「大概率不做」，仅当压测发现命中/局部渲染确有瓶颈
且产品可接受 z 序约束时再评估。

### 6.2 命中面收敛

- **`hittable={false}` 用满**：媒体 image/canvas、badge、标题/描述 text、
  装饰 rect 全部不可命中——每卡只留卡片 group（拖拽/选中）参与命中。
  这同时避免 leafer 为这些节点创建 hitCanvas。
- **port 按需挂载**：POC 的 4 个 port 常驻可命中 ellipse，改为 hover 卡片
  时才挂载 port 层（`RxUIHovered` 驱动函数 child——单卡范围的函数 child
  结构切换是合法的，它只影响一张卡，与 §3.2 禁止的全局档位切换不同）。

## 7. 媒体生命周期（视频/图片的资源预算）

### 7.1 视频

- **可见 + 档位门控**：只有「在视口内 && lod === full && 允许自动播放」的
  视频卡片才创建 `<video>` 元素并拉流；离开视口或降档立即暂停并释放，
  卡面回退到封面帧（poster 图，走图片管线）。
- **全局并发上限**：同时活跃的视频 ≤ N（建议 4–6），超出的显示封面 +
  播放按钮。用全局登记表（id 集合 + atom 计数）实现。
- **单一全局 ticker**：所有活跃视频共享一个 rAF 循环，循环里逐个
  `drawImage` + `forceRender()`，替代每卡一个 rAF。axle 侧沉淀
  `createSharedTicker()` 工具。视频帧率 ≤ 30fps，ticker 可隔帧绘制。

### 7.2 图片

- **缩略图金字塔**：按 lod 档位换 url（dot 不加载、simple 用 ~128px、
  full 用媒体区 2x ≈ 420px；原图只在双击放大时加载）。档位是挂载时固定的
  prop（§3.2），url 随行重挂载自然切换，无需响应式绑定档位。
- **解码预算队列**：集中调度 `createImageBitmap`，按「距视口中心距离」
  排优先级，每帧限量解码，快速平移时队首过期任务直接丢弃。
- **卸载释放**：卡片卸载即释放位图引用；浏览器缓存负责二次加载的网络层，
  内存只为视口内容付费。**引擎侧已验证**：leafer `ImageManager` 是引用
  计数制（`use++/use--`），无引用图片进回收列表、超过 `maxRecycled`
  阈值自动 `Resource.remove`，大图立即释放——图片内存由引擎托管且有界，
  不需要自建驱逐策略。压测时确认 `maxRecycled` / `Platform.image.maxCacheSize`
  默认值在万级缩略图浏览路径下够用即可。

## 8. 选中集与编辑器集成

虚拟化后元素会被销毁重建，一切「持有元素引用」的机制都要改为 id-based：

- **选中集是数据层的 id 集合**（`RxSet<id>` 或等价物），不持有 leafer
  元素引用。选中的卡片进入 `rxWindowedList` 的 pin 集合强制保活。
- **框选/全选在数据层做**：框选 = 空间索引范围查询（含屏外卡片），
  Ctrl+A = 全量 id；选中反馈只对挂载中的卡片渲染（选中框绑定
  `() => selected.has(card.id)`）。
- **leafer-editor 的 Editor 与虚拟化存在结构性冲突**（Editor 持元素引用画
  变换框，被选元素卸载时编辑框悬空）。Phase 3 集成时二选一：选中元素
  一律 pin（小选中集可行，「全选 10k 张再缩到 dot 档」需上限保护）；
  或不用 Editor 的元素绑定模式，选框/变换手柄基于数据层自绘。
  本方案先按前者设计 pin 上限（如 200，超出退化为只读框选反馈）。

## 9. 数据层与启动路径

- **数据加载**：10k 卡片的 model 数据（**含位置与尺寸**，见 §2.1 的
  bounds 硬约束）先行加载并建索引（网格索引 O(n)，一次性 < 10ms 量级），
  **不创建任何 Host**；首帧只挂载视口内卡片 → 可交互时间与总量无关。
- `RxList` 全量数据常驻没有问题（1 万个普通对象 + position atom 是 MB 级）。
  引擎桥（§1）只在挂载期间存在，监听成本天然 O(视口内容量)。
- 超大项目（10 万+）再考虑数据分页/按区域拉取，接口在空间索引上天然对齐
  （按格子拉取），本方案不展开。

## 10. 度量与验收

没有基线的优化不可信。配套建设：

1. **压测 playground**（`playground/stress.tsx`）：参数化生成 1k / 10k / 50k
   卡片 + 按比例的连线（图文/视频混合），复用 POC 的卡片组件。
2. **指标面板**：FPS（rAF 间隔滑动窗口）、场景图节点数、挂载/卸载次数、
   JS 堆（`performance.memory`）。
3. **验收标准**（10k 卡片，中端设备；帧指标用分位口径，避免「平均 fps +
   单帧上限」混用导致无法判定）：
   - 平移/缩放/拖拽全程：帧时长 **P95 ≤ 20ms**（≈ 55fps 体感），
     **最大帧 ≤ 100ms**（只允许出现在跨档位瞬间，且每次手势至多一帧）；
   - 拖动单卡 + 10 条关联连线：帧时长 P95 ≤ 20ms；
   - 首次可交互 < 1s（数据就绪后；fit-all 打开走底衬自绘，应趋近于零）；
   - 场景图节点数峰值 < 5,000（dot 档不逐卡挂载，该标准在全档位下自洽）；
   - 任意时刻画布不出现空白区域（未挂载区域由常驻底衬 DotLayer 显示色块，
     手势结束后 500ms 内细节补齐至缓冲区边界）；
   - 卡片叠放次序在任意浏览路径后保持稳定（z-order 回归用例）；
   - 拖拽帧上无回声写（引擎桥单测：同帧不出现第二次 `property.change`）；
   - **陈旧底衬回归用例**：程序化移动/增删一张**未挂载**卡片后，底衬
     色块在下一 rAF 内更新——专门接住 §3.3 失效机制的接线，防止该
     路径被遗漏或回归时其余验收全绿而漏检。
4. `Debug.showRepaint = true` 验证脏区不被意外放大（阴影、滤镜会扩大
   renderBounds）。

## 12. 明确不做的方向（及原因）

- **换 WebGL 渲染器**：leafer 架构上可插拔，但当前瓶颈全部在「画多少」而非
  「怎么画」。虚拟化 + LOD 后视口内节点数恒定在数百级，canvas 2d 足够；
  dot 档的万级色块由 DotLayer 单节点命令式绘制，同样不需要 GPU 管线。
  留作 leafer 引擎层未来选项，不进本方案。
- **Web Worker 渲染**：leafer 支持 worker，但跨线程后引擎桥/事件桥接复杂度
  陡增，且主线程瓶颈同样被虚拟化消解。不进本方案。
- **双向绑定视口**：维持 Phase 2 结论，视口反向操作走 `leafer.zoom()`。
- **全量 bitmap 缓存整个画布**（tile 化 minimap 除外）：缓存失效管理复杂，
  与 LOD 收益重叠。若后续 leafer 官方「光速引擎」插件成熟，届时再评估。
